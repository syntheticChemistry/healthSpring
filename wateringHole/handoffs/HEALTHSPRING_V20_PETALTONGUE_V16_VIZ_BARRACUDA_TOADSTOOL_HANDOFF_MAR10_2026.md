## Part 2: For petalTongue — Visualization Patterns to Absorb

### 2.1 V16 Scenario Node Pattern

Each V16 primitive follows a consistent builder pattern:

```
build_{domain}(s: &mut HealthScenario)
  1. Compute real math output using barraCuda CPU primitives
  2. Down-sample dense time-series to ≤500 points (display-friendly)
  3. Construct DataChannels (TimeSeries, Bar, Gauge)
  4. Attach ClinicalRange annotations for clinical thresholds
  5. Push ScenarioNode with domain-specific capabilities tag
```

| Node | DataChannels | ClinicalRanges |
|------|-------------|----------------|
| `mm_nonlinear_pk` | 3× TimeSeries (dose curves), 1× Bar (AUC vs dose), 1× Gauge (t½) | Therapeutic (10–20 mg/L), Toxic (20–40 mg/L) |
| `antibiotic_perturbation` | 1× TimeSeries (recovery curve), 1× Gauge (nadir diversity), 1× Bar (recovery%) | Healthy (>3.0), Dysbiotic (<1.5) |
| `scfa_production` | 3× TimeSeries (acetate/propionate/butyrate), 1× Bar (molar ratios), 1× Gauge (total) | Normal (80–120 mM) |
| `gut_brain_serotonin` | 1× Bar (tryptophan cascade: availability → 5-HTP → serotonin), 1× Gauge (serotonin %), 1× Bar (diversity effect) | Normal serotonin (>50%), Low (<30%) |
| `eda_stress_detection` | 1× TimeSeries (SCL), 1× TimeSeries (phasic), 1× Gauge (stress index) | Relaxed (0–0.3), Moderate (0.3–0.6), High (0.6–1.0) |
| `arrhythmia_classification` | 1× Bar (beat counts by class), 1× Scatter3D (template correlations), 1× Gauge (PVC burden) | Normal (<3% PVC), Elevated (3–10%), High (>10%) |

**petalTongue action**: These 6 node builders demonstrate a reusable pattern for any domain's scenario generation. The real-data-in, display-friendly-out pipeline could become a canonical builder trait.

### 2.2 Compute Pipeline Visualization

Four scenario builders show how to visualize non-clinical system metrics:

| Builder | Visualizes | Key DataChannels |
|---------|-----------|-----------------|
| `gpu_scaling_study()` | Batch timing vs scale (3 ops) | 3× TimeSeries, 1× Bar (Rust/Python speedup), 1× Gauge |

**petalTongue action**: System-level dashboards (hardware topology, dispatch routing, performance curves) use the same `HealthScenario` schema as clinical data. This confirms petalTongue's data model is universal across domain and infrastructure.

### 2.3 IPC Push + File Fallback

Both Exp088 and the `dump_scenarios` binary implement the same pattern:
1. Try `PetalTonguePushClient::discover()` to find a running petalTongue
2. If found, push all scenarios via JSON-RPC (`push_render` or `push_render_with_config`)
3. If not found, write JSON files to `sandbox/scenarios/` for offline inspection

Exp089 additionally uses `StreamSession` for live streaming:
- `push_initial_render(combined_scenario)` for the full patient view
- `push_pk_point(time, concentration)` for live PK curve updates
- `push_hrv_update(scfa_value)` repurposed for SCFA monitoring

**petalTongue action**: The IPC fallback-to-file pattern is proven across 3 binaries. Consider making this a first-class utility in the petalTongue client SDK.

## Part 3: For barraCuda — Visualization-Ready Math

### 3.1 Display-Friendly Output Patterns

V16 scenario builders revealed that barraCuda's math functions produce output at different granularities:

| Function | Raw Output | Display Treatment |
|----------|-----------|-------------------|
| `mm_pk_simulate` | `(Vec<f64>, Vec<f64>)` — 1000+ points at 0.01-day resolution | Down-sampled `step_by(n/500)` |
| `antibiotic_perturbation` | `Vec<(f64, f64)>` — ~100 points | Used directly (already sparse) |
| `scfa_production` | `(f64, f64, f64)` scalar | Evaluated at multiple substrate levels for curves |
| `classify_beat` | `(BeatClass, f64)` per beat | Aggregated into counts and scatter |

**barraCuda action**: Consider adding optional `resolution` or `max_points` parameters to simulation functions for direct display-ready output. This avoids each consumer re-implementing down-sampling.

### 3.2 ClinicalRange Metadata

The scenario builders attach domain-specific threshold ranges. These thresholds could live in barraCuda as constants alongside the math:

```
PHENYTOIN_THERAPEUTIC_RANGE: (10.0, 20.0) mg/L
PHENYTOIN_TOXIC_THRESHOLD: 20.0 mg/L
SCFA_NORMAL_TOTAL: (80.0, 120.0) mM
PVC_BURDEN_ELEVATED: 0.03 (3%)
```

**barraCuda action**: Consider adding clinical reference ranges as module-level constants near the corresponding math functions. This keeps domain knowledge co-located with the algorithms.

## Part 4: For toadStool — Streaming Dashboard Integration

### 4.1 StreamSession Usage Validated

Exp089 validates the full streaming pipeline:
1. `StreamSession::discover()` → finds petalTongue socket
2. `push_initial_render(scenario)` → renders combined diagnostic + V16 view
3. `push_pk_point(time, conc)` → live PK curve updates with backpressure
4. `push_hrv_update(value)` → repurposed for SCFA monitoring

**toadStool action**: The `StreamSession` API is sufficient for patient-level dashboards. When toadStool orchestrates V16 compute batches, it can feed `push_pk_point` callbacks directly from `execute_streaming` results.

### 4.2 Combined Scenario Pattern

Exp089 demonstrates merging two independent `HealthScenario` instances:
1. Build diagnostic scenario from `assess_patient` + `population_montecarlo`
2. Build V16 scenario from `v16_study()`
3. Merge nodes from both into a single scenario with combined edges

**toadStool action**: This pattern enables progressive scenario assembly — toadStool can start with a base scenario and append V16 analysis nodes as compute completes.

## Part 5: Discoveries for Upstream Evolution

### 5.1 All 7 DataChannel Types Exercised

V20 confirms that all 7 `DataChannel` types are exercised by real healthSpring data:

| DataChannel | Where Used |
|------------|-----------|
| TimeSeries | PK curves, antibiotic recovery, SCFA kinetics, EDA signals, GPU scaling |
| Distribution | Population PK Monte Carlo, endocrine distributions |
| Bar | AUC comparison, SCFA ratios, serotonin cascade, beat counts, Rust/Python speedup |
| Gauge | Half-life, nadir diversity, total SCFA, stress index, PVC burden, speedup factor |
| Spectrum | HRV power spectrum (LF/HF bands) |
| Heatmap | Antibiotic perturbation diversity matrix |
| Scatter3D | Beat template correlation space |

**Upstream recommendation**: healthSpring's V16 scenarios provide comprehensive test fixtures for petalTongue renderer development. Each `DataChannel` type has at least one real-world usage with clinically meaningful data.

### 5.2 Scenario JSON Size

Full study (34 nodes, 38 edges): ~180 KB JSON. Individual track scenarios: 5–30 KB each. V16 study: ~45 KB. These sizes are well within IPC push limits and suitable for web rendering.

### 5.3 Cross-Track Edge Pattern

The `full_study()` merger adds cross-track edges that link V16 nodes to existing tracks:
- `mm_nonlinear_pk` → `pk_one_compartment` (PKPD track)
- `antibiotic_perturbation` → `diversity_profile` (microbiome track)
- `eda_stress_detection` → `hrv_assessment` (biosignal track)
- `gut_brain_serotonin` → `testosterone_gut_axis` (endocrine track)

These demonstrate how new analysis nodes integrate with established pipelines without modifying existing scenarios.
